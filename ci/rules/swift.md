# Правила для Swift-кода (swift.md)

Документ описывает Swift‑специфичные правила, дополняющие `general.md`.  
Основан на Apple Swift API Design Guidelines и Swift Style Guide (Google и др.).

Приоритет: безопасность (опционалы, ошибки, потоки) → ясный API‑дизайн → читаемость/стиль → производительность.
Сборка и компиляция кода уже проверены в CI, не нужно поднимать это как замечание.

---

## Блокирующие

- Код соответствует базовым правилам форматирования и стилю проекта.
    - Проверяй:
        - Отступы, переносы строк, расположение скобок, длина строк.

- Безопасная работа с опционалами: минимизация `!` и `as!`.
    - Проверяй:
        - `!` и `try!` используются только там, где невозможно получить nil/исключение, и это чётко обосновано.
        - Преобразования типов `as!` заменены на безопасные `as?` с обработкой ошибок, если есть риск несоответствия.
    - Нарушение:
        - `let user = userRepository.find(id: id)!` без гарантии наличия.
        - Force unwrap UI‑элементов без гарантий жизненного цикла.

- Нет явных retain‑циклов и утечек памяти в типичных паттернах.
    - Проверяй:
        - Кложуры, хранящиеся дольше контекста (`UIView.animate`, completion‑handlers, Combine, async‑callbacks).
        - Делегаты и замыкания: использование `[weak self]`/`[unowned self]` там, где объект может быть освобождён.
    - Нарушение:
        - Захват `self` в кложуре, который живёт столько же или дольше, чем `self`, без `[weak self]`, при этом кложура хранится в свойстве.

- Асинхронный и многопоточный код корректно использует очереди/конкурентность.
    - Проверяй:
        - Обновления UI происходят на главной очереди (main thread).
        - При использовании async/await не смешиваются старые и новые API без необходимости.
        - Нет возможных гонок при доступе к общему состоянию.
    - Нарушение:
        - Обновление UI из фонового потока.
        - Запись в разделяемые mutable‑структуры из разных очередей без синхронизации.

- Ошибки не игнорируются.
    - Проверяй:
        - `try?` без логирования/обработки для важного кода.
        - Пустые `catch`‑блоки.
    - Нарушение:
        - `try? dangerousCall()` без дальнейшей проверки результата.
        - `do { try something() } catch {}`.

## Важные

- Имена и API‑дизайн следуют официальным Swift Guidelines.
    - Читаемость в месте вызова важнее краткости.
    - Имена методов/параметров отражают суть: `loadUser(id:)`, а не `do(id:)`.
    - Предпочтение глаголам для функций с побочными эффектами и существительным/прилагательным — для вычисляемых свойств.

- Активное использование `guard` для раннего выхода и уплощения вложенности.
    - `guard` вместо вложенных `if let` для проверки предусловий.
    - Улучшение читаемости, особенно в функциях с несколькими условиями.

- Предпочтение value‑типам (struct/enum) над class там, где это возможно.
    - Структуры для простых моделей данных.
    - Enum с associated values для конечных автоматов и состояний.

- Чёткий уровень доступа (`private`, `fileprivate`, `internal`, `public`, `open`).
    - Не оставлять всё `internal` по умолчанию, если реализация должна быть скрыта.
    - Публичный API минимален и документирован.

- Ресурсы корректно управляются через `defer`/RAII‑подход.
    - Файлы, блокировки, транзакции — закрытие в `defer` после успешного открытия.
    - Нет «расползания» логики закрытия по нескольким веткам.

- Структура файлов логична.
    - Один тип (или тесно связанные типы) на файл.
    - Чёткий порядок: типы, расширения, приватные helpers.

- Тесты написаны в принятом стиле (XCTest/Quick/Nimble и т.п.).
    - Покрытие основных сценариев и ошибок.
    - Использование `setUp`/`tearDown`, фикстур, given‑when‑then, если это принято в проекте.

- Использование современных возможностей языка.
    - `Result`, async/await, property wrappers, если это упрощает код.
    - Но без чрезмерной «магии», ухудшающей читаемость.

## Небольшие

- Стиль форматирования выровнен (скобки, пробелы, переносы) — по принятому Swift style guide.
- Используются trailing closures, где это улучшает читаемость, но не во вред ясности.
- Нет длинных цепочек вызовов без переносов, если это ухудшает чтение.
- «Магические числа» и строки вынесены в константы или enum.
- Комментарии информативны и не дублируют очевидный код.
- Используются стандартные коллекции и алгоритмы (`map`, `filter`, `reduce`, `compactMap`) там, где это делает код короче и понятнее.

---

## Консистентность

- Новые API выглядят «по‑свифтовски» и вписываются в существующий код:
    - те же подходы к именованию, опционалам, ошибкам.
- Для iOS/macOS:
    - Соблюдаются принятые в проекте архитектурные паттерны (MVC/MVVM/VIPER/SwiftUI‑подходы).
    - Использование DI, координаторов, менеджеров сети выровнено с текущей кодовой базой.
- Тестовая инфраструктура (XCTest‑тейсты, снапшот‑тесты, UI‑тесты) используется так же, как в существующих модулях.
- Использование сторонних библиотек (Alamofire, Combine, RxSwift, SwiftUI‑экосистема и т.п.) следует уже принятым в проекте соглашениям (инициализация, error‑handling, логирование, ретраи).
