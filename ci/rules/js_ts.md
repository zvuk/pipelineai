# Правила для JavaScript/TypeScript (js_ts.md)

Документ описывает правила для JavaScript и TypeScript, дополняющие `general.md`.  
Основан на рекомендациях Microsoft Engineering Playbook, TypeScript code review‑чеклистах, популярных стайлгайдах (Airbnb, Prettier/ESLint) и гайдлайнах по безопасному JS/TS.

Приоритет: функциональная корректность → безопасность (XSS, инъекции, зависимости) → типобезопасность (TS) → читаемость → производительность.
Сборка и компиляция кода уже проверены в CI, не нужно поднимать это как замечание.

---

## Блокирующие

- Обработка промисов корректна; нет необработанных промисов и rejected‑состояний.
    - Проверяй:
        - Все функции, возвращающие `Promise`, либо `await`‑ятся, либо явно возвращаются вверх по цепочке.
        - Нет «забытых» промисов (вызов без `await` и без `.then/.catch`).
        - В верхнем уровне потоков (HTTP‑хендлеры, cron‑jobs) есть обработка ошибок async‑кода.
    - Нарушение:
        - `doSomethingAsync();` — результат не используется.
        - Нет `.catch`/`try/catch` там, где отказ должен быть зафиксирован.

- Нет XSS и небезопасного обновления DOM.
    - Проверяй:
        - Любое использование `innerHTML`, `dangerouslySetInnerHTML` (React), `v-html` (Vue) и аналогов.
        - Шаблонные строки с пользовательскими данными, вставляемые в HTML/JS.
    - Нарушение:
        - Вставка несанитизированного пользовательского ввода в HTML.
        - Прямое использование `document.write` с внешними данными.

- Нет опасного использования `eval`, `Function`, `setTimeout`/`setInterval` со строками.
    - Проверяй:
        - Любые вызовы `eval`, `new Function`, `setTimeout("...", ...)`, особенно с данными снаружи.
    - Нарушение:
        - `eval(userCode)` или `setTimeout(userString, 1000)`.

- В серверном JS/TS (Node.js) нет опасного использования системных команд.
    - Проверяй:
        - `child_process.exec`/`execFile`/`spawn` с пользовательским вводом.
        - Прямое использование путей из внешних данных в файловых операциях без нормализации.
    - Нарушение:
        - `exec("rm -rf " + userPath)`.

- Для TypeScript: типобезопасность не отключается без причины.
    - Проверяй:
        - Использование `any`, `unknown`, `as any`, `@ts-ignore`, `@ts-expect-error`.
        - Отсутствие типов у публичных API (функций, возвращаемых значений, props/DTO).
    - Нарушение:
        - Массовое использование `any` в новом коде.
        - `@ts-ignore` поверх реальной проблемы, которую можно и нужно исправить.

## Важные

- Используются `const` и `let` вместо `var`.
    - `const` по умолчанию, `let` только если значение реально меняется.

- Модули/файлы имеют чёткую ответственность.
    - Файл не должен содержать сразу и HTTP‑хендлеры, и низкоуровневый доступ к БД, и бизнес‑логику.
    - Разделение по слоям: контроллеры, сервисы, репозитории, утилиты.

- Типы в TypeScript описаны явно и отражают доменную модель.
    - Интерфейсы/типы для DTO, запросов, ответов, конфигураций.
    - Использование union‑типов, enum/литеральных типов для ограниченных наборов значений.
    - Проверяй:
        - Соответствие типов реальным данным (особенно API‑ответам).

- Обработка ошибок в async‑коде структурирована.
    - В HTTP‑хендлерах и фоновых задачах:
        - Любая ошибка приводит к корректному ответу/логированию, а не молчаливому падению.
    - Используются централизованные обработчики ошибок (middleware, error boundary и т.п.), если они есть в проекте.

- Зависимости и импорты управляются аккуратно.
    - Не подключаются тяжёлые модули только ради одного helper’а, если в проекте есть локальный аналог.

- В React/Vue/других SPA‑фреймворках компоненты простые и переиспользуемые.
    - Большие компоненты (> ~200 строк) с несколькими обязанностями стоит разбить.
    - Локальное состояние, хранилище, эффекты (`useEffect`, watchers) используются осмысленно.

- TypeScript: минимизация `any` и правильная работа с `null`/`undefined`.
    - Использовать строго типизированные поля, включая опциональные (`foo?: string`).
    - Явно обрабатывать случаи отсутствия значения (`if (value == null) { ... }`).

- Безопасная работа с зависимостями.
    - Проверка на использование поддерживаемых версий, отсутствие очевидно уязвимых пакетов (по информации из SCA‑инструментов, если применяются).
    - Нет прямого использования «сырых» сборок из CDN/неизвестных источников в прод‑коде.

## Небольшие

- Строки форматируются через шаблонные литералы (backticks) при наличии подстановок.
- Используются современные возможности языка:
    - Optional chaining (`?.`), nullish coalescing (`??`), деструктуризация, spread/rest‑операторы.
- Нет «магических чисел»/строк — важно для статусов, типов, ролей.
- В циклах по массиву используется `for...of`/`forEach` вместо индексных циклов, если индекс не нужен.
- Для поиска/фильтрации/трансформации коллекций используются `map`, `filter`, `reduce`, если это повышает читаемость.

---

## Консистентность

- Стиль кода выровнен с уже существующими JS/TS‑модулями:
    - те же правила ESLint/Prettier, те же подходы к структуре файлов.
- В React/Vue/Angular используются те же архитектурные паттерны, что и в проекте:
    - организация компонентов, хуков, стора, роутинга.
- На server‑side (Node.js/Nest.js/Express/etc.) соблюдаются принятые паттерны:
    - структура модулей, middleware, обработка ошибок, логирование, DI.
- Соглашения по тестам (Jest, Vitest, Mocha, Cypress, Playwright и т.п.) соблюдаются:
    - структура `describe`/`it`, файлы `*.spec.ts`/`*.test.ts`, расположение фикстур.
