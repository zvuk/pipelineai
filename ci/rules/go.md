# Правила для Go-кода

Документ объединяет внутренние правила, рекомендации и лучшие практики для проверки качества Go-кода в проекте при автоматизированном code-review.
Основывается на *Go Code Review Comments*, *Effective Go*, *Google Go Style Guide* и принятых практиках проекта.

---

# 0. Общие указания для агента

1. Сначала убедиться, что фрагмент кода действительно Go.
2. При проверке опираться на:

    * правила этого документа;
    * идиомы из *Effective Go* и *CodeReviewComments*;
    * примеры из соседних файлов проекта.
3. Правило считается нарушенным только при наличии конкретной выдержки кода, которую можно процитировать.
4. Сборка и компиляция кода уже проверены в CI, не нужно поднимать это как замечание.

---

# 1. Блокирующие

## Форматирование `gofmt`

**Требования:**

* Код полностью форматирован `gofmt`.

**Примеры нарушений:**

* Нестандартные отступы, «лесенки», визуальные артефакты.
* Сложные конструкции `if/for` в одну строку.
* Скобки и блоки не соответствуют gofmt-стилю.

---

## Все ошибки проверяются

**Требования:**

* Ошибки не игнорируются (`_ = fn()`, пустые `if err != nil {}` — запрещено).
* Исключения допускаются только если прямо задокументированы в проекте.

**Агент проверяет:**

* Любой вызов, возвращающий `error`, должен приводить — либо к логированию, либо к возврату, либо к корректному завершающему действию.

**Примеры:**

* `_ = writer.Flush()`
* `if err != nil {}` без логирования и действий

---

## Запрещён `panic` в рабочем коде

**Разрешён только для:**

* невозможных инвариантов,
* инициализации, без которой программа не может стартовать,
* осознанных заглушек (`// not implemented`).

**Примеры нарушений:**

* `panic(err)` при обработке пользовательского ввода.
* `panic("should not happen")` в достижимой ветке.

---

## Нет гонок данных

**Агент проверяет:**

* Доступ к `map`, `slice`, полям `struct` из нескольких goroutine.
* Наличие защиты (`sync.Mutex`, `sync.RWMutex`, `sync.Map`).
* Наличие классического бага захвата переменной цикла:

  ```go
  for _, v := range xs {
      go func() { use(v) }() // потенциальная гонка
  }
  ```

**Признаки нарушений:**

* Запись в `m[k] = ...` и чтение `m[k]` из разных goroutine без синхронизации.
* Общий счётчик без мьютекса/атомиков.

---

## Управляемый жизненный цикл goroutine

**Требования:**

* Все долго живущие goroutine должны корректно завершаться:

    * иметь `context.Context`, либо
    * явный канал/механизм остановки.

* Горутины обязаны слушать `ctx.Done()` или закрытие канала.

**Примеры нарушений:**

* Бесконечный цикл без проверки контекста.
* `go someFn()` без возможности дождаться shutdown.

---

## Корректное закрытие ресурсов (`io.Closer` и др.)

**Требования:**

* Любой ресурс (файлы, `http.Response.Body`, `sql.Rows`, соединения) закрывается во всех ветках выполнения.
* Использовать `defer resource.Close()` там, где возможно.

**Примеры нарушений:**

* `resp, _ := client.Do(req)` без `defer resp.Body.Close()`.
* `rows, _ := db.Query()` без `rows.Close()`.

---

# 2. Важные

## Имена пакетов короткие и осмысленные

**Требования:**

* Имя пакета должно отражать назначение (`order`, `payment`, `session`).
* Избегать абстрактных имён (`util`, `helpers`, `misc`).

---

## Экспортируемые сущности документированы

**Требования:**

* Экспортируемые типы/функции имеют комментарии вида:

  ```go
  Name описывает ...
  ```

* Комментарий начинается с имени сущности.

**Пояснения для агента:**

* В Go экспортируемыми считаются только идентификаторы, имя которых начинается с **заглавной буквы**.
* Функции, типы, переменные и методы, начинающиеся со **строчной буквы**, не экспортируются из пакета и **не подпадают** под это правило.
* Перед тем как делать замечание об экспортируемых сущностях, убедись по коду, что сущность действительно экспортируется (например, функция реально видна другим пакетам).

---

## Ошибки оборачиваются с контекстом

**Требования:**

* `return err` не допускается без добавления контекста.

* Использовать:

  ```go
  fmt.Errorf("operation X failed: %w", err)
  ```

* Избегать двойного логирования (и лог, и возврат той же ошибки).

---

## Table-driven tests

**Требования:**

* Для функций, валидаций, конвертаций — использовать табличные тесты.
* Однотипные тесты объединяются в `[]struct{ ... }`.

---

## Конфигурация не хардкодится

**Требования:**

* Параметры (URL, таймауты, лимиты) читаются из конфигов, env или флагов.
* Нет «магических чисел» без пояснений.

---

## Долгоживущие структуры не удерживают большие буферы

**Требования:**

* Нельзя сохранять под-слайс большого массива — нужно делать `copy`.
* Кэши, очереди, глобальные структуры — без скрытого удержания гигантских `[]byte`.

---

## Крупные структуры передаются по указателю

**Требования:**

* Если структура тяжёлая, её передают `*T` ради снижения копирования.
* Допустимо только если не нарушает читаемость.

---

# 3. Небольшие

## Именование переменных и скоуп

* В узком скоупе допустимы короткие имена (`i`, `tt`, `v`).
* В широком — только говорящие (`userID`, `config`, `retryLimit`).

---

## Нет избыточных аллокаций

* Избегать `make([]T, 0)` там, где подходит `nil`.
* Не создавать временные объекты в горячих циклах.

---

## Логирование не зашумляет основную логику

* В горячих путях минимум логов.
* Debug-логи не должны засорять критические участки.

---

## Указание capacity для слайсов и мап

* Если размер известен заранее — всегда задавать `cap`.

---

## Не создавать долгоживущие под-слайсы

* Хранить только скопированный фрагмент (`copy`), а не часть огромного буфера.

---

## В горячих циклах избегать тяжёлых конструкций

* Не использовать `defer` в tight loop при видимой просадке.
* Закрытия — явные вызовы.

---

## Идиоматические конструкции Go

* `if err != nil { return ... }` вместо вложенных if/else.
* `for range` там, где индекс не нужен.
* Не использовать naked return в больших функциях.

---

# 4. Консистентность

## Переиспользование существующих сервисов/репозиториев

* Новая бизнес-логика использует уже существующие интерфейсы, а не дублирует.

---

## Единый подход к обработке ошибок

* Логирование, ретраи, метрики — в тех же местах и теми же механизмами, что и в других модулях проекта.

---

## Консистентность HTTP/gRPC/GraphQL-хендлеров

* Использовать те же middleware.
* Структура ответов и формат ошибок совпадают с существующими хендлерами.
* Обработка контекста — аналогично соседнему коду.
