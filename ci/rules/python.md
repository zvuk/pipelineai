# Правила для Python-кода

Документ содержи внутренние правила, которые используются при автоматизированном код-ревью Python-кода в проекте.
Опирается на PEP 8, лучшие практики Python и архитектурные требования проекта.

---

# 0. Общие указания для агента

1. Считать, что проект следует PEP 8, если нет противоположных признаков.
2. Сверяться с соседними модулями — они показывают принятый стиль (импорты, аннотации, структура).
3. Считать правило нарушенным только при наличии конкретного фрагмента кода.
4. Сборка и компиляция кода уже проверены в CI, не нужно поднимать это как замечание.

---

# 1. Блокирующие требования

## Базовое форматирование кода (PEP 8)

**Требования:**

* Отступы — 4 пробела, без смешения табов.
* Разумная длина строк (обычно до 79–99 символов).
* Пробелы вокруг операторов — по PEP 8.
* Нет «лесенок», нелогичных переносов или ручного выравнивания.

---

## Нет «голых» `except:` или неограниченного `except Exception:`

**Запрещено:**

* `except:`
* `except Exception:` без фильтрации или явного обоснования.

**Разрешено:**

* Специфические исключения (`ValueError`, `IOError` и т.д.).
* Верхний уровень воркера/CLI — если есть логирование и корректное завершение.

---

## Работа с ресурсами выполняется через контекстные менеджеры

**Требования:**

* Файлы, соединения, блокировки, сессии должны использовать `with`.
* Если `with` невозможен — ресурс закрывается во всех ветках.

**Примеры нарушений:**

* `open('file')` без `with`.
* `lock.acquire()` без гарантированного освобождения.

---

## Нет прямого `print` в рабочем коде

**Требования:**

* В прод-коде используется `logging` или проектный логгер.
* `print` допустим только в CLI/скриптах с прямым пользовательским выводом.

---

## Пользовательские данные валидируются при входе

**Требования:**

* Входные данные (HTTP/CLI/файлы) валидируются до использования.
* Используются Pydantic/DRF-сериализаторы/FormSchema или ручные проверки.
* Нет прямой передачи непроверенных данных в SQL, команды ОС, системные функции.

---

# 2. Важные требования

## Type hints для публичных функций/методов

**Требования:**

* Публичные API должны иметь аннотации аргументов и возвращаемых значений.
* Следовать общему стилю аннотирования в проекте.

---

## Архитектурная структура модулей

**Требования:**

* Доменная логика не смешивается с инфраструктурой.
* View/handler не содержат SQL/сырого доступа к БД, если есть репозитории.
* HTTP-клиенты, БД-клиенты, настройки — в соответствующих слоях.

---

## Тесты в принятом в проекте стиле

**Требования:**

* Использовать тот же фреймворк (pytest/unittest), что и проект.
* Стиль Arrange-Act-Assert.
* Правильный нейминг: `test_*.py`.
* Использовать те же фикстуры/хелперы, что и существующие тесты.

---

## Нет тяжелой логики на уровне модуля

**Запрещено:**

* Выполнение запросов к БД/HTTP при импорте модуля.
* Запуск фоновых задач.

**Разрешено:**

* Объявление констант/классов/функций.
* Создание логгера.
* Легкая инициализация.

---

## Использование `if __name__ == "__main__":` в скриптах

**Требования:**

* Все исполняемые файлы должны иметь точку входа через этот паттерн.
* Логика запуска — внутри функции (`main()`).

---

# 3. Небольшие замечания

## Единообразный стиль форматирования строк

**Требования:**

* В новом коде использовать единый подход (предпочтение f-строкам).
* Не смешивать `.format`, f-строки и `%` без причины.

---

## Списковые включения и генераторы приносят пользу читаемости

**Требования:**

* Простые преобразования коллекций использовать через comprehension.
* Не усложнять вложенными comprehension, если это снижает читаемость.

---

## Нет неиспользуемых переменных

**Требования:**

* Не оставлять переменные, которые нигде не читаются (кроме `_` по соглашению).

---

## Именование по PEP 8

**Требования:**

* `snake_case` — функции и переменные.
* `CamelCase` — классы.
* `UPPER_SNAKE_CASE` — константы.

---

# 4. Консистентность

## Новая бизнес-логика следует существующим паттернам проекта

**Требования:**

* Структура сервисов, DTO, репозиториев соответствует текущей архитектуре.
* Соблюдать интерфейсы, точки расширения, базовые классы.

---

## Новые эндпоинты соответствуют проектным стандартам API

**Требования:**

* Формат ошибок, схемы запросов/ответов, валидация — такие же, как в проекте.
* Совпадение структуры JSON/DTO, error-объектов.

---

## Использование ORM/SQL-слоя следует принятым подходам

**Требования:**

* Управление сессиями/транзакциями — как в существующем слое.
* Не создавать сырых подключений, если есть абстракции ORM.
* Не писать SQL напрямую там, где используется ORM без необходимости.
