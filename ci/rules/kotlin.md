# Правила для Kotlin-кода (kotlin.md)

Документ описывает Kotlin‑специфичные правила, дополняющие `general.md`.  
Основан на официальных Kotlin Coding Conventions, Android Kotlin Style Guide и сводных Kotlin‑стайлгайдах.

Приоритет: корректность и null‑безопасность → безопасность и корутины → читаемость и идиоматичность → производительность.
Сборка и компиляция кода уже проверены в CI, не нужно поднимать это как замечание.

---

## Блокирующие

- Код соответствует базовому форматированию и стайлгайду Kotlin.
    - Проверяй:
        - Отступы (обычно 4 пробела), переносы строк, размещение фигурных скобок.
    - Нарушение: код визуально явно не отформатирован.

- Null‑безопасность: нет необоснованных `!!` и опасных кастов.
    - Проверяй:
        - Использование `!!` — допустимо только там, где есть жёсткая гарантия не‑null (и лучше с комментарием).
        - Приведения типов через `as` вместо безопасного `as?`, приводящие к возможным `ClassCastException`.
    - Нарушение:
        - `val name = user.name!!` без проверки, что `user.name` точно не null.
        - `val v = x as SomeType` для внешних данных.

- Правильная работа с nullable/nonnull типами.
    - Проверяй:
        - Не присваивается ли `T?` в `T` без проверки.
        - Все пути кода для nullable‑значений обрабатываются (особенно при парсинге JSON/DTO).
    - Нарушение:
        - Присваивание `val id: String = dto.id` при объявлении `dto.id: String?`.

- Ресурсы, реализующие `Closeable`/`AutoCloseable`, корректно закрываются.
    - Предпочтительно использование `use { ... }`.
    - Нарушение:
        - `FileInputStream`, JDBC‑ресурсы, HTTP‑ответы без `use`/`close` во всех ветках.

- Коррутины не создают утечки и не блокируют потоки.
    - Проверяй:
        - Не используется `GlobalScope.launch`/`GlobalScope.async` в прод‑коде (кроме очень узких обоснованных случаев).
        - Внутри корутин нет блокирующих вызовов (`Thread.sleep`, блокирующий I/O) без `withContext(Dispatchers.IO)` и т.п.
        - Есть поддержка отмены через `CoroutineScope`/`Job`.
    - Нарушение:
        - Фоновая задача запускается через `GlobalScope.launch` без управления жизненным циклом.
        - В UI‑коде внутри корутины выполняется блокирующий I/O на `Dispatchers.Main`.

- Исключения не игнорируются.
    - Проверяй:
        - Пустые `catch (e: Exception)` / `catch (e: Throwable)` без логирования, rethrow или осмысленной обработки.
        - Скрытие ошибок через `runCatching { ... }` без анализа результата.
    - Нарушение:
        - `catch (e: Exception) {}`.
        - `runCatching { dangerous() }` и игнорирование `Result`.

- Для многопоточного доступа к общему состоянию используется синхронизация.
    - Проверяй:
        - Изменение `var`‑полей из разных потоков/корутин без синхронизации/мутаторов/atomic типов.
        - Использование небезопасных коллекций (`mutableListOf`, `HashMap`) из разных потоков.

## Важные

- Предпочтение `val` над `var`.
    - Всё, что можно сделать неизменяемым, должно быть `val`.
    - Меняемые поля — только если действительно требуется мутабельность.

- Имена по Kotlin‑конвенциям.
    - `camelCase` для функций и свойств, `PascalCase` для классов/интерфейсов/enum, `UPPER_SNAKE_CASE` для констант (`const val`).
    - Избегать венгерской нотации и суффиксов типа `mSomething`.

- Используются идиоматичные конструкции Kotlin.
    - `when` вместо длинных цепочек `if/else` по одному и тому же выражению.
    - Расширения (`extension functions`) для utility‑логики, связанной с конкретным типом.
    - `data class` для простых DTO/значимых объектов.
    - `sealed class`/`enum class` для закрытых иерархий состояний.

- Scoping‑функции (`let`, `apply`, `also`, `run`, `with`) используются осмысленно.
    - Не вкладывать друг в друга без необходимости (снижение читаемости).
    - Выбор scoping‑функции по назначению (`apply` для конфигурации объекта, `let` для работы с результатом и т.п.).

- Nullable‑логика читаема.
    - Предпочитать `?.` и `?:` вместо каскадов `if (x != null)`.
    - Но не превращать код в «чейнинг‑лапшу» — для сложной логики лучше обычные `if`/`when`.

- Коррутины структурированы.
    - Используются `CoroutineScope` привязанные к жизненному циклу (например, `viewModelScope`, `lifecycleScope` в Android).
    - Исключения обрабатываются (например, `CoroutineExceptionHandler` для глобальных задач).

- Используются подходящие коллекции и immutability.
    - Для возвращаемых значений и API — по возможности `List`, `Set`, `Map`, а не мутабельные варианты.
    - Внешний код не должен иметь возможность незаметно изменить внутреннее состояние модуля.

- Extension‑функции и DSL не перегружают API.
    - Расширения — для реально полезных случаев, а не для «косметики».
    - DSL должны быть ясны и не ломать привычную читаемость Kotlin.

## Небольшие

- Строки оформлены через шаблоны (`"$value"`) вместо конкатенации, где это повышает читаемость.
- Нет «магических чисел» и строк.
    - Константы вынесены в `const val`/`object`/`companion object`, если улучшается понимание.

- Порядок членов в классе и файле соответствует общим конвенциям.
    - Сначала свойства, потом конструктор/инициализация, затем методы, затем вложенные типы.

- Используется единый способ логирования (например, общий logger/обёртка), а не сырые `println`.
- Не злоупотреблять inline‑функциями; `inline` используется только там, где есть реальный выигрыш (например, generic‑DSL, no‑alloc callbacks).

---

## Консистентность

- Стиль кода выровнен с существующими Kotlin‑модулями проекта (форматирование, наименование, использование корутин, DI и т.п.).
- Для Android‑кода:
    - Следовать общим архитектурным паттернам (MVVM/MVI/Clean), используемым в проекте.
    - Соблюдать принятые подходы к навигации, DI, работе с `ViewModel` и жизненным циклом.
- Использование библиотек (Kotlinx.coroutines, Kotlinx.serialization, Ktor, Spring и др.) выровнено с тем, как они уже применяются в кодовой базе.
- Тесты на Kotlin (JUnit, Kotest, MockK и др.) пишутся в одном стиле с существующими тестовыми модулями.
